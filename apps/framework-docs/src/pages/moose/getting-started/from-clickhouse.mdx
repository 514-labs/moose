## Quick Setup (5 minutes)

<Steps>

### Get Your ClickHouse Connection

You need a ClickHouse connection URL in this format:

```
http://username:password@host:port/?database=database_name
```

<ToggleBlock openText="Need help finding your connection details?" closeText="Hide credential help">
<Tabs items={["ClickHouse Cloud", "Self-Hosted", "Docker", "Playground"]}>
<Tabs.Tab>
1. Log into your [ClickHouse Cloud console](https://clickhouse.cloud/)
2. Go to your service details page
3. Find "Connect" or "Connection Details" section
4. Copy the HTTPS endpoint and your username/password
</Tabs.Tab>
<Tabs.Tab>
- Check your ClickHouse config file (usually `/etc/clickhouse-server/config.xml`)
- Look for `<http_port>` (default: 8123) and `<https_port>` (default: 8443)
- Check users config in `/etc/clickhouse-server/users.xml` or users.d/ directory
- Default user is often `default` with no password
</Tabs.Tab>
<Tabs.Tab>
- Check your docker-compose.yml or docker run command for environment variables
- Look for `CLICKHOUSE_USER`, `CLICKHOUSE_PASSWORD`, `CLICKHOUSE_DB`
- Default is usually `http://default:@localhost:8123/?database=default`
</Tabs.Tab>
<Tabs.Tab>
Want to test without your own ClickHouse? Use the [ClickHouse Playground](https://clickhouse.com/docs/getting-started/playground):

```txt copy
https://explorer:@play.clickhouse.com:443/?database=default
```
</Tabs.Tab>
</Tabs>
</ToggleBlock>

### 1. Seed Local Development Data

</Callout>

### 2. Build Materialized Views in Code

Stop managing materialized views with manual DDL. Define them in code:

<TypeScript>
```typescript filename="Smart aggregations that don't break" copy
import { userEvents } from "./tables";
import { MaterializedView, sql } from "@514labs/moose-lib";

interface DailyUserStats {
  user_id: number;
  date: Date;
  event_count: number;
}

const dailyStats = new MaterializedView<DailyUserStats>({
  selectStatement: sql`
    SELECT 
      ${userEvents.user_id}, 
      toDate(${userEvents.timestamp}) as date, 
      count(*) as event_count
    FROM ${userEvents} 
    GROUP BY ${userEvents.user_id}, date
  `,
  selectTables: [userEvents],
  tableName: "daily_user_stats",
  materializedViewName: "daily_user_stats_mv",
  orderByFields: ["user_id", "date"]
});

// ✅ Moose handles all the DDL, dependencies, and migrations
```
</TypeScript>

<Python>
```python filename="Smart aggregations that don't break" copy
from moose_lib import MaterializedView, MaterializedViewOptions
from app.tables import user_events

class DailyUserStats(BaseModel):
    user_id: int
    date: date  
    event_count: int

daily_stats = MaterializedView[DailyUserStats](
    MaterializedViewOptions(
        select_statement="""
            SELECT user_id, toDate(timestamp) as date, count(*) as event_count
            FROM user_events GROUP BY user_id, date
        """,
        select_tables=[user_events],
        table_name="daily_user_stats",
        materialized_view_name="daily_user_stats_mv",
        order_by_fields=["user_id", "date"]
    )
)

# ✅ Moose handles all the DDL, dependencies, and migrations
```
</Python>

### 3. Generate APIs in Minutes

Turn your ClickHouse queries into production APIs:

<TypeScript>
```typescript filename="Production-ready APIs" copy
import { ConsumptionApi } from "@514labs/moose-lib";

interface GetUserStatsParams {
  userId: number;
  startDate: string;
  endDate: string;
}

export const UserStatsApi = new ConsumptionApi<GetUserStatsParams, DailyUserStats[]>(
  "user-stats",
  async ({ userId, startDate, endDate }, { client, sql }) => {
    const query = sql`
      SELECT date, event_count 
      FROM daily_user_stats 
      WHERE user_id = ${userId} 
        AND date BETWEEN ${startDate} AND ${endDate}
      ORDER BY date
    `;
    
    const data = await client.query.execute<DailyUserStats>(query);
    return await data.json();
  }
);

// ✅ Auto-generated OpenAPI docs, validation, and type safety
```
</TypeScript>

<Python>
```python filename="Production-ready APIs" copy
from moose_lib import ConsumptionApi, MooseClient

class GetUserStatsParams(BaseModel):
    user_id: int
    start_date: date
    end_date: date

def get_user_stats(client: MooseClient, params: GetUserStatsParams):
    query = f"""
        SELECT date, event_count 
        FROM daily_user_stats 
        WHERE user_id = {params.user_id} 
          AND date BETWEEN '{params.start_date}' AND '{params.end_date}'
        ORDER BY date
    """
    return client.query.execute(query)

api = ConsumptionApi[GetUserStatsParams, DailyUserStats](
    name="user-stats", 
    query_function=get_user_stats
)

# ✅ Auto-generated OpenAPI docs, validation, and type safety
```
</Python>

### 4. Add Real-Time Streaming (Optional)

Optionally add streaming capabilities to your existing tables:

<TypeScript>
```typescript filename="Add streaming to existing tables" copy
import { Stream } from "@514labs/moose-lib";
import { userEvents } from "./tables";

// Add streaming layer to your existing ClickHouse table
export const userEventsStream = new Stream<UserEvents>("user_events", {
  destination: userEvents  // Points to your existing ClickHouse table
});

// ✅ New data can now stream in real-time to your existing table
```
</TypeScript>

<Python>
```python filename="Add streaming to existing tables" copy
from moose_lib import Stream, StreamConfig
from app.tables import user_events, UserEvents

# Add streaming layer to your existing ClickHouse table
user_events_stream = Stream[UserEvents]("user_events", StreamConfig(
    destination=user_events  # Points to your existing ClickHouse table
))

# ✅ New data can now stream in real-time to your existing table
```
</Python>

<Callout type="info" title="Non-destructive addition">
This creates a streaming layer that writes to your existing ClickHouse table. Your current data ingestion methods continue to work alongside the new streaming capabilities.
</Callout>

## Known Limitations

<Callout type="warning" title="Tables with these types will be skipped">
Right now, tables featuring these types will be skipped during the initialization process.
</Callout>

### Currently Unsupported Types:
- **`Map(K, V)` types**
- **`Tuple(T1, T2, ...)` types**  
- **`Geo` types** (Point, Ring, Polygon, etc.)
- **`Enum` types with empty string keys**

### Partial Support:
- **`JSON` types** - <TypeScript inline>Maps to `Record<string, any>`</TypeScript><Python inline>Maps to `any`</Python>
- **`LowCardinality(T)`** - Maps to base type `T` (optimization annotations available)

### What we're working on:
- **Selective table import** (currently imports all tables)
- **Default value annotations** 
- **Advanced ClickHouse type support**
