# Workflow Orchestration

<LanguageSwitcher />

## Overview

Moose workflows enable developers to automate sequences of tasks in a maintainable, reliable way. A workflow is a series of tasks that execute in order, with support for complex task dependencies and error handling.

<Callout type="info" title="Workflow API">
Moose workflows use explicit `Task` and `Workflow` classes for powerful task orchestration capabilities with type safety and flexible task composition. This replaces the previous file-based routing approach with dynamic, programmatic task orchestration.
</Callout>

<Callout type="info" title="Powered by Temporal">
This workflow abstraction is powered by Temporal under the hood. You can use the Temporal GUI to monitor your workflow runs as they execute, providing extra debugging capabilities.
</Callout>

## Key Improvements

The new workflow API provides several advantages over the previous file-based approach:

- **Dynamic Routing**: Tasks can be composed and chained programmatically rather than relying on static file structure
- **Type Safety**: Full TypeScript/Python type checking for task inputs and outputs
- **Explicit Dependencies**: Clear task relationships defined in code rather than inferred from file names
- **Better Composition**: Tasks can be reused across multiple workflows
- **Simplified Structure**: No need to maintain specific file naming conventions

## Quickstart

To create a new workflow, run the following command:


## Creating Tasks

<TypeScript>

### Task Definition

Tasks are defined using the `Task` class with typed input/output interfaces:

```typescript filename="app/main.ts" copy
import { Task, Workflow } from "@514labs/moose-lib";

interface FirstTaskInput {
    name: string;
}

interface FirstTaskOutput {
    name: string;
    greeting: string;
    counter: number;
}

// Define the task
const firstTask = new Task<FirstTaskInput, FirstTaskOutput>("first", {
    run: async (input) => {
        const name = input.name ?? "world";
        const greeting = `hello, ${name}!`;
        
        return {
            name: name,
            greeting: greeting,
            counter: 1
        };
    },
    timeout: "5m",
    retries: 3
});
```

### Task Configuration

Each task can be configured with:

- **`run`**: The async function that performs the task logic
- **`onComplete`**: Array of tasks to execute after this task completes successfully
- **`timeout`**: Maximum execution time for the task (e.g., "5m", "1h")
- **`retries`**: Number of retry attempts if the task fails

</TypeScript>

<Python>

### Task Definition

Tasks are defined using the `Task` class with typed input/output interfaces:

```python filename="app/main.py" copy
from moose_lib import Task, Workflow, TaskConfig, WorkflowConfig
from pydantic import BaseModel

class FirstTaskInput(BaseModel):
    name: str

class FirstTaskOutput(BaseModel):
    name: str
    greeting: str
    counter: int

# Define task handler
async def first_handler(input: FirstTaskInput) -> FirstTaskOutput:
    name = input.name or "world"
    greeting = f"hello, {name}!"
    
    return FirstTaskOutput(
        name=name,
        greeting=greeting,
        counter=1
    )

# Define the task
first_task = Task[FirstTaskInput, FirstTaskOutput](
    name="first",
    config=TaskConfig(
        run=first_handler,
        timeout="5m",
        retries=3
    )
)
```

### Task Configuration

Each task can be configured with:

- **`run`**: The async function that performs the task logic
- **`on_complete`**: List of tasks to execute after this task completes successfully
- **`timeout`**: Maximum execution time for the task (e.g., "5m", "1h")
- **`retries`**: Number of retry attempts if the task fails

</Python>

## Creating Workflows

<TypeScript>

### Workflow Definition

Workflows orchestrate multiple tasks and define their execution order:

```typescript filename="app/main.ts" copy
import { Task, Workflow } from "@514labs/moose-lib";

// ... task definitions ...

// Configure task dependencies
firstTask.config.onComplete = [secondTask];

// Create the workflow
const exampleWorkflow = new Workflow("example", {
    startingTask: firstTask,
    timeout: "1h",
    retries: 3,
    schedule: "0 12 * * *" // Optional: cron expression for scheduling
});
```

### Workflow Configuration

Workflows are configured with:

- **`startingTask`**: The first task to execute in the workflow
- **`timeout`**: Maximum execution time for the entire workflow
- **`retries`**: Number of retry attempts if the workflow fails
- **`schedule`**: Optional cron expression for scheduled execution

</TypeScript>

## Task Chaining and Data Flow

<TypeScript>

### Dynamic Task Composition

The new API enables dynamic task composition where tasks can be chained programmatically:

```typescript filename="app/main.ts" copy
import { Task, Workflow } from "@514labs/moose-lib";

interface TaskAOutput {
    userId: number;
    userName: string;
}

interface TaskBInput {
    userId: number;
    userName: string;
}

interface TaskBOutput {
    userId: number;
    userName: string;
    processedAt: string;
}

const taskA = new Task<{}, TaskAOutput>("taskA", {
    run: async () => {
        return {
            userId: 123,
            userName: "john_doe"
        };
    }
});

const taskB = new Task<TaskBInput, TaskBOutput>("taskB", {
    run: async (input) => {
        // input automatically has the correct type from TaskA's output
        return {
            userId: input.userId,
            userName: input.userName,
            processedAt: new Date().toISOString()
        };
    }
});

// Dynamic task chaining - can be conditional or computed
taskA.config.onComplete = [taskB];

const workflow = new Workflow("dataFlow", {
    startingTask: taskA
});
```

The TypeScript compiler ensures type safety between task inputs and outputs, catching mismatches at compile time.

</TypeScript>

<Python>

### Dynamic Task Composition

The new API enables dynamic task composition where tasks can be chained programmatically:

```python filename="app/main.py" copy
from moose_lib import Task, Workflow, TaskConfig, WorkflowConfig
from pydantic import BaseModel
from datetime import datetime

class TaskAOutput(BaseModel):
    user_id: int
    user_name: str

class TaskBInput(BaseModel):
    user_id: int
    user_name: str

class TaskBOutput(BaseModel):
    user_id: int
    user_name: str
    processed_at: str

async def task_a_handler() -> TaskAOutput:
    return TaskAOutput(
        user_id=123,
        user_name="john_doe"
    )

async def task_b_handler(input: TaskBInput) -> TaskBOutput:
    # input automatically has the correct type from TaskA's output
    return TaskBOutput(
        user_id=input.user_id,
        user_name=input.user_name,
        processed_at=datetime.now().isoformat()
    )

task_a = Task[dict, TaskAOutput](
    name="taskA",
    config=TaskConfig(run=task_a_handler)
)

task_b = Task[TaskBInput, TaskBOutput](
    name="taskB", 
    config=TaskConfig(run=task_b_handler)
)

# Dynamic task chaining - can be conditional or computed
task_a.config.on_complete = [task_b]

workflow = Workflow(
    name="dataFlow",
    config=WorkflowConfig(starting_task=task_a)
)
```

Pydantic ensures type safety between task inputs and outputs, catching mismatches at runtime.

</Python>

## Running Workflows

As you develop workflows, you can run them directly using the Moose CLI:

```bash filename="Terminal" copy
moose workflow run example
```
The terminal will print the following:
```txt filename="Terminal"
      Workflow 'example' started successfully.
View it in the Temporal dashboard: http://localhost:8080/namespaces/default/workflows/example/3a1cc066-33bf-49ce-8671-63ecdcb72f2a/history
```

Notice that you are given a URL to view the workflow in the Temporal dashboard. This is a helpful way to monitor the workflow execution in real time and debug any issues through a GUI.

### Passing Input to Workflows
When you run a workflow, you can pass input to the workflow by using the `--input` flag.

```bash filename="Terminal" copy
moose workflow run example --input '{"name": "John"}'
```

The input is passed to the workflow as a JSON string.

## Debugging Workflows
While the Temporal dashboard is a helpful tool for debugging, you can also leverage the Moose CLI to monitor and debug workflows. This is useful if you want to monitor a workflow without having to leave your terminal.

Use the `moose workflow status` command to monitor a workflow:

```bash filename="Terminal" copy
moose workflow status example
```

## Complete Example

<TypeScript>

Here's a complete workflow example that demonstrates task chaining, error handling, and scheduling:

```typescript filename="app/main.ts" copy
import { Task, Workflow } from "@514labs/moose-lib";

// Data models
interface UserData {
    userId: number;
    email: string;
}

interface ProcessedUser {
    userId: number;
    email: string;
    processedAt: string;
    status: string;
}

interface NotificationResult {
    userId: number;
    notificationSent: boolean;
    timestamp: string;
}

// Task 1: Fetch user data
const fetchUserTask = new Task<{}, UserData>("fetchUser", {
    run: async () => {
        // Simulate API call
        return {
            userId: 123,
            email: "user@example.com"
        };
    },
    timeout: "30s",
    retries: 3
});

// Task 2: Process user data
const processUserTask = new Task<UserData, ProcessedUser>("processUser", {
    run: async (input) => {
        return {
            userId: input.userId,
            email: input.email,
            processedAt: new Date().toISOString(),
            status: "processed"
        };
    },
    timeout: "1m",
    retries: 2
});

// Task 3: Send notification
const notifyUserTask = new Task<ProcessedUser, NotificationResult>("notifyUser", {
    run: async (input) => {
        // Simulate notification sending
        return {
            userId: input.userId,
            notificationSent: true,
            timestamp: new Date().toISOString()
        };
    },
    timeout: "30s",
    retries: 1
});

// Configure task chain
fetchUserTask.config.onComplete = [processUserTask];
processUserTask.config.onComplete = [notifyUserTask];

// Create workflow
const userProcessingWorkflow = new Workflow("userProcessing", {
    startingTask: fetchUserTask,
    timeout: "10m",
    retries: 2,
    schedule: "0 */6 * * *" // Run every 6 hours
});

export { userProcessingWorkflow };
```

</TypeScript>

<Python>

Here's a complete workflow example that demonstrates task chaining, error handling, and scheduling:

```python filename="app/main.py" copy

    processed_at: str
    status: str

    user_id: int
    notification_sent: bool
    timestamp: str

# Task handlers
async def fetch_user_handler() -> UserData:
    # Simulate API call
    return UserData(
        user_id=123,
        email="user@example.com"
    )

    return ProcessedUser(
        user_id=input.user_id,
        email=input.email,

        notification_sent=True,

    name="fetchUser",

        run=fetch_user_handler,
        timeout="30s",
        retries=3

)

    name="processUser",
```

## Triggering Workflows from APIs

You can trigger workflows from your consumption APIs to create event-driven workflows:

<TypeScript>
```typescript filename="app/apis/trigger_workflow.ts" copy
import { ConsumptionApi } from "@514labs/moose-lib";

interface WorkflowParams {
    inputValue: string;
}

const triggerApi = new ConsumptionApi<WorkflowParams>("trigger-workflow", ({inputValue}: WorkflowParams, {client, sql}) => {
    client.workflows.execute(workflow="example", params={inputValue: inputValue})
})
```
</TypeScript>

<Python>
```python filename="app/apis/trigger_workflow.py" copy
from moose_lib import MooseClient, ConsumptionApi
from pydantic import BaseModel, Field

class WorkflowParams(BaseModel):
    input_value: str = Field(default="default")

def run(client: MooseClient, params: WorkflowParams) -> dict:
    return client.workflows.execute(
        workflow="example",
        params={input_value: params.input_value}
    )

api = ConsumptionApi[WorkflowParams]("trigger-workflow", run)
```

For more information on how to set up a Moose API, see the [Consumption API](/moose/building/consumption-apis) documentation.
</Python>

### Task-Level Errors and Retries

For more granular control over task-level errors and retries, you can configure your individual tasks to have their own retry behavior:

<TypeScript>

Configure task-level retries and timeouts in the task configuration:

```typescript filename="app/main.ts" copy
import { Task } from "@514labs/moose-lib";

const resilientTask = new Task<InputType, OutputType>("resilient", {
    run: async (input) => {
        // Task logic that might fail
        throw new Error("This is a test error");
        return { result: "success" };
    },
    retries: 5,      // Task-specific retry count
    timeout: "10m"   // Task-specific timeout
});
```

</TypeScript>

<Python>

Configure task-level retries and timeouts in the task configuration:


### Example: Workflow and Task Retry Interplay

When configuring retries, it's important to understand how workflow-level and task-level retries interact. Consider the following scenario:

<TypeScript>

```typescript filename="app/main.ts" copy
import { Task, Workflow } from "@514labs/moose-lib";

// Task with 3 retries
const failingTask = new Task<{}, {}>("failing", {
    run: async () => {
        throw new Error("This is a test error");
        return {};
    },
    retries: 3
});

// Workflow with 2 retries
const workflow = new Workflow("example", {
    startingTask: failingTask,
    retries: 2
});
```

</TypeScript>

<Python>

```python filename="app/main.py" copy
from moose_lib import Task, Workflow, TaskConfig, WorkflowConfig

async def failing_handler() -> dict:
    raise Exception("This is a test error")
    return {}

# Task with 3 retries
failing_task = Task[dict, dict](
    name="failing",
    config=TaskConfig(
        run=failing_handler,
        retries=3
    )
)

# Workflow with 2 retries
workflow = Workflow(
    name="example",
    config=WorkflowConfig(
        starting_task=failing_task,
        retries=2
    )
)
```

</Python>

If the execution of the workflow encounters an error, the retry sequence would proceed as follows:

1. **Workflow Attempt 1**
   - **Task Attempt 1**: Task fails
   - **Task Attempt 2**: Task fails
   - **Task Attempt 3**: Task fails
   - Workflow attempt fails after exhausting task retries

2. **Workflow Attempt 2**
   - **Task Attempt 1**: Task fails
   - **Task Attempt 2**: Task fails
   - **Task Attempt 3**: Task fails
   - Workflow attempt fails after exhausting task retries

In this example, the workflow will make a total of 2 attempts, and each task within those attempts will retry up to 3 times before the workflow itself retries.

### Terminating Workflows

To terminate a workflow before it has finished running, use the `workflow terminate` command.

```bash filename="Terminal" copy

```

<Callout type="warning" title="Terminating Workflows">
You cannot run the same workflow concurrently. Use the `terminate` command to stop the workflow before triggering it again.
</Callout>

## Best Practices

### 1. Type Safety
Always define clear input/output interfaces for your tasks to leverage compile-time type checking.

### 2. Task Granularity
Keep tasks focused on single responsibilities. This makes them more reusable and easier to test.

### 3. Error Handling
Configure appropriate timeouts and retries based on your task's expected behavior and failure modes.

### 4. Dynamic Composition
Take advantage of the dynamic routing capabilities to create flexible, reusable workflows.

### 5. Monitoring
Use the Temporal dashboard to monitor workflow execution and debug issues.

## Migration from File-Based Workflows

If you're migrating from the previous file-based workflow approach:

### What Changed

**Previous Approach (V1)**:
- **TypeScript**: Static file-based routing with implicit task dependencies
- **Python**: `@task` decorator with static file-based routing
- Task execution order determined by file naming conventions
- Limited flexibility in task composition

**New Approach**:
- **Both Languages**: Explicit `Task` and `Workflow` classes
- Dynamic, programmatic task composition
- Type-safe task chaining
- Flexible task reuse across workflows

### Migration Steps

1. **Replace file-based structure** with explicit `Task` and `Workflow` definitions
2. **Define type interfaces** for task inputs and outputs
3. **Configure task dependencies** using `onComplete`/`on_complete`
4. **Move all definitions** to `app/main.ts` or `app/main.py`
5. **Test thoroughly** to ensure task chaining works as expected

The new API provides much more flexibility and type safety while maintaining the same underlying Temporal orchestration capabilities.
