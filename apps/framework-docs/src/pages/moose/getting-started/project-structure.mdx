# Project Organization

<Callout type="info" title="Moose Project Structure: Only What You Need">
Moose is designed to be flexible. There are only a few hard requirements for how your project must be structured. Everything else is up to you!
</Callout>

<FileTree>

Each example above includes:
- An `app/` directory (required)
- A root export file (`index.ts` for TypeScript, `main.py` for Python)
- A `moose.config.toml` at the project root

You can add as many files as you want inside `app/`—just make sure to export/import what you want Moose to deploy from your root file.

<Callout type="info" title="What is the .moose folder?">
The `.moose/` folder is where Moose maintains some files it needs for its internals. The most relevant file for you as a developer is the generated `openapi.yaml` spec, which describes your ingest and consumption APIs. You can use this file to explore or integrate your APIs with tools like Swagger UI or Postman.
</Callout>

Here are a few recommended project structures for typical Moose use cases, each using a different organizational pattern:

## Everything Else is Flexible

- **Directory Names:** Use the recommended `ingest/`, `views/`, `apis/`, etc., or create your own folders and groupings.
- **File Layout:** Organize by domain, by component type, or any pattern that fits your team/project.
- **Depth:** Keep it flat or nest as needed—just avoid deep, confusing hierarchies.

## How to Apply Your Own Patterns

- **Group by domain:**  
  `app/users/ingest.py`, `app/products/views.py`, etc.
- **Group by type:**  
  `app/ingest/users.py`, `app/views/products.py`, etc.
- **Single file for small projects:**  
  Just put everything in <TypeScript inline>index.ts</TypeScript>/<Python inline>main.py</Python>.

## Strategies for Organizing Your Project

Here are a few high-level strategies you might consider:

- **By Domain (Vertical Slices):**
  - Group files by business domain or feature area.
  - Example: `app/users/ingest.ts`, `app/products/views.ts`
- **By Component Type (Horizontal Layers):**
  - Group files by their role in the pipeline (ingest, views, apis, etc.).
  - Example: `app/ingest/users.ts`, `app/views/products.ts`
- **By Feature:**
  - Each feature or product area gets its own folder with all related files inside.
  - Example: `app/analytics/`, `app/monitoring/`
- **Flat Structure for Small Projects:**
  - Keep everything in a few files in the `app/` directory.
  - Example: `app/index.ts`, `app/ingest.ts`, `app/views.ts`, `app/apis.ts`

You can mix and match these strategies as your project grows. The key is to keep things clear and maintainable for your team.

<Callout type="warning" title="Common Mistakes to Avoid">
- Forgetting to export/import components from your root file (`index.ts`/`main.py`)
- Deeply nested or overly complex directory structures
- Mixing unrelated domains or features in the same files
- Creating circular dependencies between components
</Callout>

- Have a `moose.config.toml` at the root
